최근 읽은 글 중에서 꽤 인상 깊었던 글이 있어 핵심 내용을 한국어로 정리해본다.

원문은 Eniko Fox의 `Software occlusion culling in Block Game`이며, 블록 게임 특성에 맞춰 CPU에서 직접 깊이 기반 오클루전 컬링을 구현한 과정을 다룬다.

원문 링크:

- <https://enikofox.com/posts/software-rendered-occlusion-culling-in-block-game/>

## 왜 CPU로 오클루전 컬링을 했나

글 작성자는 강한 GPU 환경이 아니고, 목표 하드웨어도 낮은 사양을 상정하고 있었다.  
또한 게임이 블록 기반이라 지형과 청크가 축 정렬된 큐브로 구성되어 있어, 비교적 단순한 형태의 소프트웨어 컬링이 잘 맞는 상황이었다.

핵심 판단은 이렇다.

- 동굴, 지하 공간처럼 가려진 기하가 매우 많다.
- 블록 게임은 가리는 오브젝트와 가려지는 오브젝트가 대부분 큐브다.
- GPU readback 기반 깊이 재사용은 환경상 부담이 있다.
- 그래서 낮은 해상도의 CPU depth buffer를 직접 만들어 컬링하는 쪽이 더 단순하고 실용적이다.

결과적으로 이 방식은 꽤 잘 먹혔다고 한다.

- 60FPS 기준 반 프레임 정도 안에서 동작
- 프러스텀 컬링 이후 살아남은 청크 중 최소 50% 이상 제거
- 실내나 동굴에서는 95% 이상 제거되는 경우도 있음

## 기본 개념

아이디어 자체는 단순하다.

1. 먼저 장면의 가림 정보 역할을 하는 depth buffer를 만든다.
2. 각 후보 청크가 그 depth buffer 뒤에 완전히 숨어 있는지 검사한다.
3. 조금이라도 보이면 렌더링하고, 전부 가려지면 버린다.

일반적으로는 GPU 기반 Hi-Z, previous frame 재사용, async readback 같은 고급 기법을 붙일 수 있지만, 이 글에서는 그런 복잡도를 피하고 CPU에서 직접 구현한다.

## 이 글의 최종 구현 방식

작성자는 `256x128` 해상도의 매우 작은 depth buffer를 CPU 메모리 상의 float 배열로 만들었다.  
그리고 블록 게임이라는 특성을 이용해 삼각형이 아니라 큐브 단위로 depth를 채운다.

### 1. 청크를 오클루더 계층으로 만든다

청크 크기는 `16x16x16` 이고, 이를 여러 단계의 서브청크로 다룬다.

- 레벨 0: 청크 전체
- 레벨 1: `2x2x2` 분할, 각 셀은 `8x8x8` 블록
- 레벨 2: `4x4x4` 분할, 각 셀은 `4x4x4` 블록
- 레벨 3: `8x8x8` 분할, 각 셀은 `2x2x2` 블록
- 레벨 4: `16x16x16` 분할, 각 셀은 블록 1개

레벨 4에서 먼저 `완전히 불투명하고 visible face가 있는 블록`을 오클루더 후보로 표시한다.  
그다음 상위 레벨로 올라가며 `2x2x2` 영역이 전부 불투명하고, 그 안에 실제 visible face를 가진 하위 요소가 있을 때 상위 서브청크를 오클루더로 만든다.

즉, 멀리서는 큰 덩어리로 가리고 가까이서는 작은 블록 단위로 정밀하게 가리는 구조다.

### 2. 후보 청크와 오클루더를 모은다

카메라가 갱신되면 먼저:

- visible face나 엔티티가 없는 청크를 제외
- 월드 공간에서 프러스텀 컬링
- 남은 청크를 오클루전 컬링 후보로 등록

오클루더는 거리별로 다른 레벨을 사용한다.

- 가장 가까운 영역은 블록 단위 오클루더 사용
- 더 멀어질수록 더 큰 서브청크 사용

이후에는 각 청크의 실제 데이터에 다시 접근하지 않도록, 위치/크기/index만 따로 모아 백그라운드 스레드로 넘긴다.  
이렇게 해서 thread-safe하게 오클루전 컬링을 돌린다.

### 3. CPU에서 큐브를 depth buffer에 그린다

이 구현에서 흥미로운 부분은 “삼각형 래스터라이즈” 대신 “큐브의 화면 범위를 아주 싸게 계산”한다는 점이다.

오클루더에 대해:

- 먼저 서브청크 단위로 다시 프러스텀 컬링
- 8개 코너를 스크린 좌표와 depth로 변환
- 각 y 행마다 최소 x, 최대 x를 기록
- 그 범위를 depth buffer에 수평 채우기

또 하나의 포인트는 depth를 일반적인 투영 공간 z 대신 `linear view depth`로 다룬다는 점이다.  
텍스처링이나 정밀한 GPU 파이프라인 재현이 목적이 아니므로, 더 싼 계산으로 충분하다고 본 것이다.

### 4. 오클루전 후보는 반대로 검사한다

오클루더는 “가장 먼 깊이”를 buffer에 채우고,  
후보 청크는 “가장 가까운 깊이” 기준으로 검사한다.

후보 청크가 차지하는 픽셀 중 하나라도 buffer보다 앞에 있으면 보이는 것으로 간주한다.  
어느 픽셀도 통과하지 못하면 그 청크는 완전히 가려졌다고 보고 컬링한다.

단, 후보 청크의 코너 중 일부가 near plane 뒤에 있으면 보수적으로 `visible` 처리한다.  
검사가 불완전할 때 잘못 버리는 false positive를 피하기 위한 선택이다.

## 성능 최적화 과정

이 글이 특히 좋은 이유는 최종 해법만 보여주는 게 아니라, 어디서 시간이 줄었는지를 단계별로 보여준다는 점이다.

### 1. 더 싼 좌표 변환

처음에는 일반적인 `4x4 projection matrix` 기반 계산을 썼는데, 이를:

- view rotation
- additive translation
- focal length 기반 투영
- linear depth 사용

방식으로 바꿨다.

결과:

- 곱셈 수를 크게 줄임
- 처리 시간이 대략 `150ms -> 99ms`

### 2. 월드 공간 프러스텀 컬링

스크린 공간에서 컬링하려고 하면 투영 후 비선형성 때문에 보수적으로 계산해야 해서 오히려 비효율적이었다.  
이를 월드 공간의 bounding sphere + frustum plane 검사로 바꾸면서:

- 더 싸고
- 더 정확해졌다

결과:

- 대략 `150ms -> 61ms` 수준까지 개선

### 3. visible face 없는 것 제거

처음에는 공기만 아니면 후보로 보던 기준이 너무 느슨했다.  
실제로 렌더될 수 있으려면 블록이 아니라 `visible face`가 있어야 한다.

이 기준을 적용하자:

- 후보 수가 크게 줄고
- 시간도 `34ms` 수준까지 감소

### 4. 가까운 오클루더, 먼 후보 검사

원근 때문에:

- 가까운 오클루더는 화면에서 크게 보임
- 먼 후보는 픽셀 수가 적어 검사 비용이 낮음

그래서 가까운 것 위주로 가리고, 먼 것 위주로 검사하는 전략을 쓰자 약간 더 좋아졌다.

- `37ms -> 29ms`

### 5. 멀티 레벨 서브청크

결정타는 여기다.

- 가까운 곳은 작은 오클루더
- 먼 곳은 큰 오클루더

를 사용하는 사실상의 mipmap 비슷한 구조를 도입했다.

또한 오클루더는 서브청크 단위로 세분화하되,  
오클루전 후보는 굳이 잘게 쪼개지 않고 `청크 전체`로 검사했다.

이 최적화 후:

- `29ms -> 4ms`

정도로 떨어졌다고 한다.  
글 전체에서 가장 중요한 아이디어는 사실상 이 부분이라고 봐도 된다.

## 왜 1픽셀 inset을 주는가

낮은 해상도 depth buffer를 쓰면, 실제 화면에서는 보이는 작은 틈이 버퍼에서는 막혀 보이는 경우가 생긴다.  
그러면 false occlusion이 발생한다.

이를 줄이기 위해 작성자는 오클루더를 가로/세로 1픽셀씩 줄여서 그린다.  
정밀도는 약간 잃지만, 잘못 컬링하는 문제를 크게 줄일 수 있다.

이 선택은 블록 게임처럼 큰 덩어리의 가림이 많은 장르에서는 꽤 현실적인 트레이드오프다.

## 한계점

좋은 방법이지만 만능은 아니다.

### 1. 근처에 좋은 오클루더가 없으면 성능이 떨어진다

가장 강력한 오클루더는 1블록 단위인데, 이건 멀리까지 많이 쓰기 어렵다.  
또 1픽셀 inset 때문에 멀어지면 금방 한두 픽셀, 나중에는 사실상 사라진다.

그래서:

- 공중에서 멀리 내려다보는 상황
- 주변 20블록 내에 불투명 블록이 거의 없는 상황

에서는 컬링 효율이 급격히 나빠질 수 있다.

### 2. 얇은 플레이어 구조물에는 약할 수 있다

플레이어가 만든 구조물은 1~2블록 두께인 경우가 많다.  
이런 구조는 큰 서브청크 오클루더로 승격되지 못해 멀리서 잘 안 가릴 수 있다.

### 3. 청크 경계 틈

오클루더를 청크 내부 단위로만 만들기 때문에, 1픽셀 inset과 결합되면 청크 사이 경계에 틈이 남는다.  
그 틈을 따라 원래는 가려져야 할 청크가 살아남을 수 있다.

다만 글 작성자는 실제 성능 악화는 크지 않았다고 본다.

## 앞으로의 개선 아이디어

글 마지막에 나온 후속 아이디어도 흥미롭다.

- 큐브형 서브청크 대신 축 방향 얇은 slice 오클루더 사용
  - 예: `4x4x1`, `4x1x4`, `1x4x4`
- 더 높은 렌더 거리에서는 여러 청크를 묶는 superchunk 고려
- 필요하다면 SIMD 최적화 추가

특히 얇은 slice 오클루더는 플레이어 구조물이나 지표면 같은 형태에 더 잘 맞을 가능성이 커 보인다.

## 개인적으로 중요하게 본 포인트

이 글에서 배울 점은 단순히 “오클루전 컬링을 했다”가 아니다.

### 1. 문제 구조에 맞는 단순한 해법을 택했다

GPU Hi-Z, async readback 같은 정석적이고 화려한 방법 대신:

- 블록 게임
- 큐브 기반
- 낮은 사양 목표
- CPU에서도 감당 가능한 해상도

라는 조건에 맞춰 해법을 아주 실용적으로 선택했다.

### 2. 후보와 오클루더를 다르게 설계했다

오클루더는 세분화하고, 후보는 청크 전체로 둔 선택이 매우 좋다.  
이 구분이 없으면 검사 비용이 너무 쉽게 폭증한다.

### 3. 멀티 레벨 구조가 핵심이다

결국 이 시스템의 성패는 “거리별로 다른 크기의 오클루더를 쓰는가”에 달려 있다.  
가까운 곳은 정밀도, 먼 곳은 개수 절감이라는 요구를 동시에 만족시킨다.

## 짧은 결론

이 글은 블록/복셀 게임에서 소프트웨어 오클루전 컬링을 고민할 때 매우 참고할 만하다.

- 구현 아이디어가 단순하다
- 최적화 과정이 단계별로 잘 드러난다
- 왜 빨라졌는지가 설명된다
- 한계와 실패 케이스도 솔직하게 정리돼 있다

특히 `CPU depth buffer + 거리별 서브청크 오클루더 + 보수적인 visible 판정` 조합은,  
GPU 기능을 적극 쓰기 어려운 엔진이나 저사양 타깃 프로젝트에서 꽤 현실적인 선택지로 보인다.
